# 附加实验:硬件看门狗(IWDG)+ 自动回滚闭环

> 目标:用 STM32 独立看门狗(IWDG)把"坏固件**卡死**"也纳入自动回滚的保护范围。
> 之前的回滚只能应对"坏固件**主动复位**",本实验补上"卡死不复位"这一半——这才是企业级 health check 的完整形态。

## 1. 原理回顾

IWDG 是一个**独立时钟(LSI)驱动的递减计数器**,数到 0 强制复位芯片。程序必须周期性"喂狗"(重装计数器)证明自己活着。卡死的代码喂不了狗 → 超时 → 复位。配合现有的"试运行标志"机制:

```
坏固件卡死 → 喂不到狗 → IWDG 复位 → 进 Boot → 发现试运行标志还在 → 回滚
```

**关键连带点**:IWDG 一旦启动,**任何复位都不会停止它**(只有断电重上电才复位 IWDG)。所以一旦 App 启用了 IWDG,**Bootloader 在备份/搬运(耗时几秒)时也必须喂狗**,否则升级过程本身会被狗咬断。本项目已在三处喂狗:App 主循环、Boot 备份/搬运循环、Boot 主循环(救砖停留时)。

## 2. 代码改动(已完成,无需手改)

| 文件 | 改动 |
|---|---|
| `Common/ota.h` | 新增 `OTA_FEED_WDG()` 宏(裸写 `IWDG->KR=0xAAAA`,App/Boot 通用) |
| `App/Core/Src/main.c` | 主循环喂狗;`SIMULATE_HANG` 演示宏(卡死坏固件) |
| `Common/ota_boot.c` | 备份/搬运循环内喂狗 |
| `Bootloader/Core/Src/main.c` | 主循环喂狗(救砖停留时) |

唯一需要你做的是用 **CubeMX 给 App 启用 IWDG**(启动看门狗),以及重新编译重烧。

## 3. CubeMX 配置(只动 App 工程)

1. 打开 `App\App.ioc`;
2. 左侧 **System Core → IWDG** → 勾选 **Activated**;
3. 切到下方 **Parameter Settings**:
   - **IWDG counter clock prescaler** = `64`
   - **IWDG down-counter reload value** = `1875`
   - 超时 = 1875 × 64 / 40000 ≈ **3 秒**(LSI 标称 40kHz)
4.(可选,强烈建议)解决调试时被复位:左侧 **System Core → SYS**,或在调试配置里冻结 IWDG;若用代码,可在 `USER CODE BEGIN 2` 加 `__HAL_DBGMCU_FREEZE_IWDG();`。不调试可忽略。
5. **GENERATE CODE**。CubeMX 会生成 `iwdg.c` 并自动 `MX_IWDG_Init()` 启动看门狗,USER CODE 区的喂狗代码不会丢。

> Bootloader **不用**在 CubeMX 配 IWDG——它不启动看门狗,只用裸寄存器喂已被 App 启动的狗。

## 4. 编译与烧录

1. **App**:CubeMX 生成后直接 F7 编译(`iwdg.c` 已自动入工程);
2. **Bootloader**:也改了(加喂狗),F7 编译后 **ST-Link 重烧 Boot**;
3. 先确保一个**正常版本**(不带 SIMULATE_HANG)已在板上运行。

## 5. 演示一:证明看门狗真的在工作

正常固件应当**持续运行、LED 不停闪、串口不重复打印启动信息**(因为主循环一直喂狗)。

要亲眼看到"狗在咬":临时把 App 主循环里的 `OTA_FEED_WDG();` 注释掉,编译烧录 → 板子会**每 3 秒复位一次**,串口反复打印 `[App] Firmware ... running`。看完把注释恢复。这一步直观证明:不喂狗 = 周期性复位。

## 6. 演示二:卡死固件被自动回滚(实验高潮)

这次不需要手动按两次复位了——IWDG 会自动产生"那次复位"。

1. **制作卡死坏固件**:Keil → Options for Target → C/C++ → **Preprocessor Symbols → Define** 里加 `SIMULATE_HANG`;`app_version.h` 改成 9.9 便于识别;编译出 bin(**不要 ST-Link 烧**);
2. **推送**(签名正常):
   ```powershell
   python ota_host.py COM7 ..\App\MDK-ARM\App\App.bin
   ```
3. **复位一次**,观察串口全自动完成:
   ```
   [Boot] Update request: v9.9 ...
   [Boot] Backing up current App ...        ← Boot 搬运期间也在喂狗,不会被打断
   [Boot] Update OK, trial boot. App must confirm.
   [App] !! SIMULATE_HANG: dead loop ...     ← 坏固件卡死,不喂狗
   (约 3 秒后 IWDG 自动复位)
   [Boot] Last FW v9.9 was NOT confirmed.
   [Boot] Rolling back from backup ...
   [Boot] Rollback OK.
   [App] Firmware v3.4 running.              ← 自动回到旧版本
   ```
4. **演示完恢复**:Keil 删掉 `SIMULATE_HANG` define、版本改回,重新编译,推送一个正常固件覆盖。

**和之前回滚演示的关键区别**:之前坏固件是"注释掉 ConfirmBoot",靠你**手动按第二次复位**模拟崩溃才触发回滚;现在 IWDG 把"卡死→复位"自动化了,真正实现**无人值守的自愈**——这正是远程 OTA 设备必须具备的能力(现场没人能去按复位键)。

## 7. 验收标准

- [ ] 正常固件持续运行不复位(喂狗有效);
- [ ] 注释喂狗后每 3 秒复位一次(看门狗确实在工作);
- [ ] `SIMULATE_HANG` 固件升级后卡死,约 3 秒被 IWDG 复位并**自动回滚**到旧版本,全程不碰复位键;
- [ ] 升级搬运过程(Boot 几秒)不被看门狗打断;
- [ ] 按住 PB0 长时间停留 Boot 不被复位(救砖路径喂狗有效)。

## 8. 常见坑

| 现象 | 原因 |
|---|---|
| 升级搬运中途板子复位、升级失败 | Boot 没喂狗(确认 ota_boot.c / Boot main.c 的喂狗已生效、Boot 已重烧) |
| 救砖停留 Boot 时每 3 秒复位 | Boot 主循环没喂狗 |
| 调试单步时总被复位 | 没冻结调试态 IWDG(见 3.4) |
| 正常 App 偶尔无故复位 | 某次主循环迭代 > 3 秒没喂狗(如超大固件接收),适当调大 reload |
| 改了喂狗超时不生效 | IWDG 配置只在复位后加载;需断电重上电或复位 |

## 9. 喂狗位置的深浅(面试加分)

本项目在**主循环**喂狗,能防"整个程序卡死"。但有个经典陷阱:如果只是**某个子任务**卡了、而主循环还在空转,狗照样被喂,看门狗就失效了。更严谨的做法是"**多任务标志位**":每个关键任务跑完置一个标志,喂狗前检查所有标志都置位才喂。这是看门狗从"防死机"进阶到"防功能异常"的关键,可作为延伸优化。
