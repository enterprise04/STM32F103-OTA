## 1. 协议设计(了解即可,代码已实现)

**上位机 → 板,统一帧格式:**

```
| 0xAA | 0x55 | CMD(1) | SEQ(1) | LEN(2,小端) | DATA(LEN) | CRC32(4,小端) |
```

- CRC32 覆盖 `CMD..DATA`,算法与 Python `zlib.crc32` 一致(板端 `crc32_calc` 实现相同多项式);
- DATA 每包固定 1024 字节(末包除外)→ 写入始终对齐 W25Q64 的 4K 扇区边界;
- 三种命令:`START`(固件大小+整包CRC+版本)→ `DATA`×N → `END`(板端整包校验+写参数区)。

**板 → 上位机,固定 4 字节应答:** `| 0x55 | 0xAA | STATUS | SEQ |`,STATUS=0 表示 OK,非 0 时上位机自动重传(最多 5 次)。

**可靠性机制:** 逐帧 CRC、停等式 ACK、超时重传、重复包识别(SEQ 回退一格直接补 ACK 不重写)、END 时对 W25Q64 里整包固件再算一次 CRC。

## 2. CubeMX:给 App 工程加 USART1 RX DMA

⚠️ 只改 **App.ioc**,Bootloader 不动。

1. 双击打开 `App\App.ioc`;
2. 左侧 **Connectivity → USART1** → 中间面板选 **DMA Settings** 标签 → 点 **Add**;
3. 下拉框选 **USART1_RX**,选中后下方参数区确认:
   - Channel:`DMA1 Channel 5`(自动)
   - Direction:`Peripheral To Memory`
   - Mode:**Normal**(不要选 Circular)
   - Data Width:两边都 `Byte`
4. 切到 **NVIC Settings** 标签确认:`DMA1 channel5 global interrupt` 已自动勾选,`USART1 global interrupt` 保持勾选(阶段①已勾);
5. 点 **GENERATE CODE**,Keil 弹出 reload 提示选 Yes。USER CODE 区的代码不会丢失。

## 6. 验收标准

上位机输出:

```
固件: ..\App\MDK-ARM\App\App.bin
大小: xxxxx 字节, CRC32: 0xXXXXXXXX, 版本: v1.1
握手 ... OK
发送 [####################] 100%  (NN/NN 包)
等待板端校验 ... OK
升级包推送完成,耗时 x.xs。板端已写入升级标志,复位后生效。
```

验收后**重新打开串口助手**按一次复位,板端会先打印 Boot/App 正常启动信息——此时跳回的还是旧固件(搬运是阶段④的事),但 W25Q64 参数区已有升级标志等着 Bootloader 来消费。传输过程中板载 LED 每收一包翻转一次,肉眼可见"数据在流动"。

## 7. 常见坑

| 现象 | 原因 |
|---|---|
| 握手超时 | 串口助手没关(占用 COM 口);或 App 没在跑(看 LED 是否慢闪) |
| 握手返回"状态/参数非法" | bin 超过 48KB(App 分区上限),或 bin 路径给错了 |
| 频繁"帧CRC错"重传 | 杜邦线接触不良/过长,降波特率试(脚本 `--baud 9600` + CubeMX 同步改) |
| 编译报 hdmarx 相关错误 | CubeMX 的 DMA 没配,或没重新生成代码 |
| 每包都"序号错" | 板子中途复位过,重跑一次脚本即可(START 会重置状态机) |

## 8. 设计要点(面试常问)

- **为什么传输期间板端不 printf?** 协议和日志共用 USART1,打印会污染应答字节流,上位机可能把日志里的字节误当成应答头。所以板端只在传输结束后打印总结。
- **为什么擦写放主循环而不是中断回调?** 扇区擦除最长 400ms,在 ISR 里做会饿死 SysTick 等其他中断;回调里只置标志,符合"中断要短"的原则。
- **为什么关 DMA 半满中断(HT)?** `ReceiveToIdle_DMA` 默认半满也回调一次,一帧会被劈成两段;关掉后只在"空闲线检测"(一帧结束)时回调一次。
- **断电续传在哪?** 当前没做——传输中断后重跑脚本即可整包重来(48KB 只要几秒)。真正的断点续传放在参数区记录 offset 即可,可作为加分扩展。
