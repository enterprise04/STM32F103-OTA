# 阶段⑤ 操作指南:数字签名(保证固件合法性)

> 前置:四阶段已完结。本阶段在 **App 工程**加入 ECDSA P-256 签名验证,只有用你的私钥签发的固件才会被设备接受。
> 验签位置:**App 端**(挡住一切经串口/OTA 通道的非法固件)。

## 1. 解决什么问题

CRC32 只能防"传输出错",**防不了伪造**——攻击者可以构造一个恶意固件并配上正确的 CRC32,设备照收。数字签名解决的是**真实性/合法性**:只有持有私钥的你能签发,任何人没有私钥都无法伪造一个能通过验证的固件。

CRC 与签名分工不同,**两者都保留**:

| | 防什么 | 谁能伪造 |
|---|---|---|
| CRC32 | 传输错误(完整性) | 任何人都能算 |
| 数字签名 | 伪造/篡改(真实性) | 只有私钥持有者 |

## 2. 原理与链路

- **算法**:ECDSA over P-256(secp256r1)+ SHA-256。M3 资源受限,RSA 库太大,ECC 是合理选择(签名/公钥各 64 字节)。
- **非对称**:私钥只在你 PC 上(签名用,**保密**),公钥编译进固件(验签用,**可公开**)。
- **流程**:
  ```
  上位机:SHA-256(固件) → 私钥 ECDSA 签名 → 64 字节签名随 START 帧下发
  设备端:收完固件 → 重算 SHA-256 → 内置公钥 uECC_verify(hash, 签名)
          通过 → 写升级请求;失败 → 拒绝(应答状态 0x06)
  ```

## 3. 新增文件

| 文件 | 说明 |
|---|---|
| `Common/sha256.c/.h` | SHA-256 实现(与 Python hashlib 一致) |
| `Common/micro-ecc/` | ECDSA 库(BSD,已配置为纯 C + 仅 secp256r1) |
| `App/Core/Inc/ecc_pubkey.h` | **公钥**,由 keygen 生成,编译进固件 |
| `Host/keygen.py` | 生成密钥对 + 导出公钥头 |
| `Host/private_key.pem` | **私钥**,只在本机,已 gitignore,**绝不上传** |

## 4. Keil 操作(只动 App 工程)

`Common` 组 → Add Existing Files,加入 **2 个 .c**:

- `..\..\Common\sha256.c`
- `..\..\Common\micro-ecc\uECC.c`

> - **include path 不用改**:`..\..\Common` 阶段①已加(覆盖 `sha256.h` 和 `micro-ecc/uECC.h`);`ecc_pubkey.h` 在 App 默认的 `..\Core\Inc`。
> - **Bootloader 工程不动**(本方案 App 端验签)。
> - **体积**:App 区 48KB,新增约 7KB,余量充足,不必调优化等级。

## 5. 密钥已生成(了解即可)

`python keygen.py` 已经生成了 `private_key.pem` 和 `ecc_pubkey.h`。
**只要不重新生成密钥,这步不用再做。** 若将来重新生成密钥(`keygen.py` 删掉 pem 后重跑),必须重新编译 App(因为公钥变了)。

## 6. 推送用法(命令不变,自动签名)

```powershell
pip install ecdsa          # 已装
cd Host
python ota_host.py COM7 ..\App\MDK-ARM\App\App.bin
```

脚本会自动读私钥、对固件签名、随握手下发。输出多一行 `握手(含签名) ... OK`,板端打印 `CRC+Sign OK`。

## 7. 验收

**① 正常固件能升级**(签名合法):
```
握手(含签名) ... OK
发送 [####################] 100%
等待板端校验 ... OK
```
板端串口:`[OTA] FW vX.Y received: ... CRC+Sign OK.`

**② 非法固件被拒绝**(签名篡改演示):
```powershell
python ota_host.py COM7 ..\App\MDK-ARM\App\App.bin --tamper
```
`--tamper` 故意破坏 1 bit 签名,模拟伪造/篡改。预期:上位机报 `签名验证失败(固件非法或被篡改)`,重试 5 次后失败退出;**板端不写升级标志**,复位仍是旧固件。这证明:没有正确签名的固件,设备坚决不收。

## 8. 安全边界(面试要会讲)

- **App 端验签的有效范围**:出厂首版 App 由 ST-Link 可信烧录,之后每次 OTA 都验签 → 信任链完整,**通过串口/OTA 通道无法刷入非法固件**。
- **超出此威胁模型的**:物理攻击(拆机直接刷 SPI Flash、用 ST-Link 重烧)。要防这些需 **Bootloader 端验签 + 芯片 Secure Boot + Flash 加密**——这是本项目作为单 MCU 教学项目的合理边界。
- 想再进一步:把验签移到 Bootloader(真正信任根,连被篡改的 App 也拦),代价是 Boot 增大可能要扩分区——已在交流中评估过。

## 9. 常见坑

| 现象 | 原因 |
|---|---|
| 编译报 `g_ecc_pubkey` 未定义 | 没运行 keygen 生成 `ecc_pubkey.h`,或 include path 缺 `..\Core\Inc` |
| uECC.c 报内联汇编错误 | 用的不是改过的 uECC.h(顶部需有 `uECC_PLATFORM = uECC_arch_other`) |
| 没篡改却报"签名验证失败" | `ecc_pubkey.h` 与当前 `private_key.pem` 不是一对 → 重跑 keygen 并**重新编译 App** |
| 验签很慢(几百 ms) | 正常,P-256 软件验签在 72MHz M3 上约 100~300ms,升级不频繁可接受 |

## 10. 面试讲法

- **CRC vs 签名**:一个防传错(完整性),一个防伪造(真实性),不可互相替代。
- **为什么 ECC 不用 RSA**:M3 资源受限,ECC 签名/公钥小(64B vs RSA-2048 的 256B)、库更小。
- **非对称原理**:私钥签、公钥验,公钥可随固件公开,私钥泄露才是灾难(所以 gitignore)。
- **威胁模型分层**:App 端验签防通道攻击;完整信任根要 Boot 端验签 + 硅级 Secure Boot。这套是 secure boot 的上半部分。
